# Журнал инженерных решений

Эта заметка нужна, чтобы через полгода не спрашивать: “почему мы сделали именно так?”

## PySide6 Вместо tkinter

Причина:

- нужен нормальный desktop GUI;
- нужна масштабируемость;
- проще делать меню, таблицы, вкладки, worker threads, localization.

Решение:

- `eis_qt.py` — active GUI target;
- tkinter/customtkinter не возвращать.

## Ядро отдельно от интерфейса

Причина:

- научная логика должна тестироваться отдельно;
- GUI не должен владеть fit model;
- будущий Chem Suite сможет переиспользовать core.

Решение:

- fit logic в `eis_core.py`;
- parser logic в `eis_io.py`;
- GUI вызывает их как service layer.

## galvani Для BioLogic

Причина:

- не писать бинарный `.mpr` parser руками;
- использовать существующую библиотеку electrochem-data ecosystem.

Tradeoff:

- GPL-3.0-or-later;
- `.mpr` EIS всё равно надо валидировать на реальном lab file.

## BIC вместо одной только ошибки фитинга

Причина:

- raw fit error любит overfit;
- сложные модели почти всегда могут улучшить residual;
- нужен penalty за число параметров.

Решение:

- best model = lowest BIC among non-BAD candidates.

## Расширенный режим скрыт по умолчанию

Причина:

- обычному пользователю не нужны bounds и manual circuit;
- advanced controls пугают и создают ошибочные решения.

Решение:

- default workflow: open -> auto-fit -> inspect -> export;
- Pro mode: presets/manual/bounds/local presets.

## Экспорт Не Локализуется

Причина:

- CSV/XLSX должны быть стабильными для скриптов и шаблонов;
- язык интерфейса не должен ломать data contract.

Решение:

- UI переводится;
- export columns/sheet names/circuit strings остаются стабильными.

## Пользовательские пресеты не хранятся в репозитории

Причина:

- bounds и схемы могут быть личными lab habits;
- нельзя случайно коммитить user config.

Решение:

- `%APPDATA%\EIS Solver\pro_presets.json`;
- fallback `.eis_solver_user/pro_presets.json`;
- fallback folder в `.gitignore`.

## Крайние масштабы не блокируют первый SPICE-экспорт

Причина:

- два крайних идеальных RC не проходят ошибку рациональной аппроксимации;
- реальные ECM в двух независимых корпусах прошли `12/12`;
- ворота уже закрывают неподдерживаемый случай без выдачи ложной модели.

Решение:

- не задерживать однофайловый пользовательский экспорт;
- масштабы `10⁻⁴ Ω` и `10⁵–10⁶ Ω` оставить известным ограничением;
- расширение масштабирования выполнять позже, не ослабляя текущие ворота.

## SPICE выдаётся атомарным пакетом

Причина:

- один `.lib` без научной модели, полосы и ошибок невозможно безопасно
  интерпретировать;
- частично записанный результат нельзя выдавать за проверенную модель;
- случайная перезапись инженерного результата разрушает воспроизводимость.

Решение:

- команда принимает только один входной спектр;
- ngspice обязателен;
- выдаётся каталог с `model.lib` и `passport.json`;
- существующий каталог не перезаписывается;
- конечный каталог появляется только после полной успешной записи.
- в GUI действие находится только в панели инструментов Pro mode, скрыто в
  обычном режиме и не дублируется в главном меню.

## Контроллер получает сразу float32 и Q31

Причина:

- `float32` проще встроить и проверить;
- STM32F103 не имеет аппаратного блока плавающей точки, поэтому для
  ограниченного времени расчёта нужен целочисленный вариант;
- два независимых подбора могли бы незаметно дать разные физические модели.

Решение:

- один дискретный набор RC-состояний порождает оба варианта C;
- `float32` принимает амперы и возвращает вольты;
- Q31 принимает и возвращает нормированные величины Q1.31;
- произведения Q31 считаются в `int64_t`, результат округляется и насыщается;
- пакет содержит общие эталонные векторы и строгий паспорт;
- верхняя частота по умолчанию ограничена `1%` частоты расчёта;
- существенный последовательный индуктивный член приводит к отказу, а не к
  неявной неверной дискретизации.

## Экспорт C остаётся специализированным действием

Причина:

- период расчёта и полный масштаб тока требуют понимания целевой прошивки;
- неверный масштаб может создать формально корректную, но непригодную модель;
- обычный пользователь анализа EIS не должен видеть контроллерные параметры.

Решение:

- действие находится только в конце панели инструментов расширенного режима;
- вне расширенного режима скрыты и действие, и общий разделитель;
- командная строка требует явные `--sample-period-us` и
  `--current-full-scale-a`.

## Версия 1 начинается с номера 0.9.0

Причина:

- функциональный объём уже близок к версии 1;
- чистая Windows-проверка и пользовательская приёмка ещё не закрыты;
- номер `1.0.0` должен означать пройденные выпускные ворота, а не намерение.

Решение:

- единая версия хранится в `eis_version.py`;
- GUI и CLI показывают `0.9.0`;
- канал выпуска — кандидат на версию 1;
- переход к `1.0.0` запрещён до независимой проверки на чистой машине и
  выбора режима распространения.

## GUI и CLI используют один автоматический подбор

Причина:

- CLI уже использовал двухступенчатый `adaptive_v2`;
- GUI напрямую перебирал полный список из 17 схем;
- одинаковая кнопка «автоматический подбор» не должна означать разные
  научные процедуры.

Решение:

- общий `fit_spectrum()` находится в `eis_pipeline.py`;
- автоматический GUI и CLI передают `circuits=None`;
- ручной ввод и выбранные наборы передают явный список;
- бюджеты, маршрутизация, второй уровень по остаткам и начальные зерна
  определяются общим слоем;
- GUI сохраняет полный `circuit_routing` в метаданных случая.

## Приёмка является одной исполняемой командой

Причина:

- отдельные ручные дымовые проверки легко забыть;
- запуск окна не доказывает чтение данных и работу подбора;
- собранная программа может потерять модуль с отложенным импортом.

Решение:

- `eis_release_check.py` проверяет тесты, GUI, CLI, SPICE, C и собранную
  программу;
- собранный exe имеет скрытый функциональный ключ `--release-smoke`;
- проверка выполняет настоящий KK и подбор, а не только ждёт живой процесс;
- BioLogic MPT проверяется отдельно, чтобы гарантировать наличие `galvani`.

## Проект выпускается под GPL-3.0-or-later

Причина:

- `galvani` распространяется под GPL-3.0-or-later;
- PySide6 имеет условия LGPL/GPL либо коммерческого лицензирования;
- владелец выбрал открытое распространение проекта под GPLv3 или более
  поздней версией.

Решение:

- сборка `0.9.0` остаётся кандидатом до чистой Windows-проверки и
  пользовательской приёмки;
- в неё включаются метаданные лицензий и `THIRD_PARTY_NOTICES.md`;
- в репозитории и сборке находятся `LICENSE`, `CITATION.cff` и
  `CITATION.md`;
- `PyMuPDF` вынесен из рабочих зависимостей в отдельный список для
  документации и не входит в приложение;
- распространение выполняется под `GPL-3.0-or-later`; закрытая проприетарная
  сборка не является выбранным режимом проекта.
